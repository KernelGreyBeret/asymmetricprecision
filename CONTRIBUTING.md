# Contributing to KGB Studio

    KGB Studio is an owner-directed, source-available development environment. Public visibility does not make the project open source.

The maintainer may accept narrowly scoped defect fixes, compatibility improvements, accessibility fixes, documentation corrections, or other changes aligned with the published architecture and roadmap. New modules, architectural changes, format changes, branding changes, and major feature work should not be submitted without prior written approval.

    ## Before submitting work

    Open an issue before preparing any substantial change. Describe the problem, the proposed scope, and the files likely to change. Do not assume that public source visibility authorizes reuse or independent development from project material.

    ## Contributions that may be considered

    - Reproducible bug fixes
    - Browser or device compatibility fixes
    - Accessibility improvements
    - Security hardening
    - Broken-link and typo corrections
    - Focused documentation corrections
    - Small performance or reliability improvements that preserve intended behavior

    ## Contributions that will normally be declined

    - Unrequested major features or redesigns
    - Changes to branding, names, characters, stories, published arguments, frameworks, or visual identity
    - Bulk automated, low-context, or speculative changes
    - AI-generated submissions that the contributor cannot fully explain, verify, and legally license
    - Contributions containing copied, scraped, proprietary, confidential, or incompatibly licensed material
    - Changes that introduce tracking, advertising, unnecessary dependencies, external services, or telemetry without prior approval

    ## Contribution process

    1. Open or identify an issue.
    2. Receive maintainer approval for substantial work.
    3. Fork the repository and create a focused branch.
    4. Make the smallest complete change necessary.
    5. Test the affected experience on relevant desktop and mobile browsers.
    6. Preserve existing notices, architecture, naming, and project direction.
    7. Submit a pull request using the repository template.
    8. Clearly disclose:
       - what changed and why;
       - how it was tested;
       - any dependencies or third-party material;
       - whether AI-assisted tools materially generated the contribution; and
       - any known limitations.

    ## Rights and contributor certification

    By submitting a contribution, you certify that:

    - you created it or otherwise have sufficient rights to submit it;
    - it does not knowingly infringe another party’s rights;
    - it contains no confidential, classified, controlled, private, or employer-owned material that you lack authority to disclose; and
    - you grant the project owner the contribution license stated in `LICENSE`.

    Submission does not guarantee acceptance. The maintainer may edit, combine, decline, remove, or later replace accepted work.

    ## Conduct

    Participation is governed by `CODE_OF_CONDUCT.md`.

    ## Security issues

    Do not report vulnerabilities in a public issue. Follow `SECURITY.md`.
