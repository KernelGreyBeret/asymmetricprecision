# AP v2 Sprint 28 — Availability / Dead-End Elimination

## Principle

A missing node should still guide the learner.

## Why this sprint exists

AP v2 includes books, frameworks, laboratories, essays, journeys, and Atlas nodes that may be complete, in development, planned, or intentionally future-facing.

A normal site treats unavailable material as a dead end.

AP cannot.

If a resource is not ready yet, the system should explain its status, why the node matters, and where the learner should go next.

## What changed

- Added an AP availability data map.
- Added status-aware book/resource surfaces.
- Added path-forward panels for planned and in-development resources.
- Added mailto actions for asking about resource availability.
- Added Atlas and journey routing from unavailable nodes.
- Added card-level availability notes on dynamic grids.
- Hardened styling so status/action surfaces do not overflow their containers.

## Design rule

Unavailable does not mean useless.

A planned book, future laboratory, or unpublished framework can still orient the learner if the system explains its role and routes them to useful adjacent material.
