# AP Naming

Version 1.0

---

## Why Naming Matters

Names shape understanding.

If our philosophy is based on helping people build better mental models, then our own language must reflect that philosophy.

Names are part of the design.

They are never placeholders.

---

## Principle

Every name should teach.

Every name should have a reason.

If we cannot explain why something is named the way it is, it should probably be renamed.

---

## Intentional All the Way Down

There is no layer of AP that is exempt from intentionality.

Not the homepage.

Not the CSS.

Not the JavaScript.

Not the folder structure.

Not the metadata.

Not the URLs.

Not the APIs.

Not the commit messages.

Everything communicates.

Everything teaches.

## Repository Philosophy

The repository itself should teach.

Someone exploring AP for the first time should begin understanding the philosophy simply by reading the source tree.

The architecture should communicate the architecture.

The system should explain itself.

---

## The Test

Ask one question.

"Does this name help someone understand the system?"

If yes...

Keep it.

If no...

Rename it.
